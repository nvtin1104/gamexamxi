KEY FILES DOCUMENTED
====================

1. MIDDLEWARE: auth.ts
   Reads token from Authorization (Bearer) or X-Auth-Token header
   Verifies JWT signature against JWT_SECRET env var
   Checks session exists in KV_SESSIONS (allows logout)
   Returns 401 if missing/invalid/expired
   Sets userId in context for downstream routes

2. MIDDLEWARE: ratelimit.ts
   IP-based sliding window rate limiting
   Routes with custom limits:
     - /api/auth/login: 10 req/min
     - /api/auth/register: 5 req/min
     - default: 100 req/min
   Skips if KV_RATELIMIT not configured
   Returns 429 if exceeded

3. MIDDLEWARE: adminAuth.ts
   Requires authenticated user (from auth middleware)
   Reads ADMIN_EMAILS env var (comma-separated)
   If empty/unset: allows all authenticated users (dev mode)
   If set: enforces whitelist, returns 403 if not admin

4. MAIN: index.ts
   Hono.js v4 application
   CORS: gamexamxi.pages.dev, localhost:3000/3001/8788
   Public routes: /api/auth, /api/health
   Protected routes: /api/users, /api/games, /api/groups, /api/shop, /api/quests
   Admin routes: /api/admin/* (auth + adminAuth)
   WebSocket: /ws/game/:eventId, /ws/group/:groupId
   Queue consumers: points, achievements, notifications

5. ROUTE: auth.ts (partial shown - 80 lines)
   POST /api/auth/register
     Input: username (3-20 chars [a-zA-Z0-9_]), email, password (8+ chars)
     Validates no duplicate email/username
     Hashes password with scrypt
     Creates user + sends WELCOME_BONUS to queue
     Returns JWT token + user
   
   POST /api/auth/login
     Input: email, password
     Verifies credentials
     Returns JWT token + user

6. ROUTE: admin.ts (complete)
   GET /api/admin/stats
     Returns: totalUsers, totalEvents, totalGroups, totalTransactions, recentUsers
   
   GET /api/admin/users?limit=20&offset=0&search=
     Paginated user list with optional email/username search
     Max 100 per page
   
   GET /api/admin/users/:id
     User detail + last 20 transactions
   
   POST /api/admin/users/:id/points
     Grant or deduct points
     Types: ADMIN_GRANT, ADMIN_DEDUCT
     Prevents negative balance
   
   GET /api/admin/events?limit=20&offset=0&status=
     Paginated events, optional status filter
   
   DELETE /api/admin/events/:id
     Hard delete event
   
   GET /api/admin/groups?limit=20&offset=0
     Paginated groups list
   
   GET /api/admin/transactions?limit=30&offset=0
     Paginated transaction history

ARCHITECTURAL PATTERNS
======================

1. Type Safety
   - All routes: Env, Variables generics
   - Zod validation on request bodies
   - Drizzle ORM for type-safe database queries

2. Middleware Chain
   - CORS -> Logger -> Rate Limit -> (optional Auth) -> (optional Admin Auth) -> Route

3. Database Access
   - All routes use createDb(c.env.DB) for Drizzle instance
   - Queries use drizzle-orm syntax (eq, like, or, sql)

4. Session Management
   - JWT tokens in Authorization header
   - Sessions stored in KV_SESSIONS for logout capability
   - 30-day TTL on sessions

5. Rate Limiting
   - Per-IP, per-route granular control
   - Custom windows and limits per endpoint
   - X-RateLimit-Remaining/Limit headers in response

6. Admin Access Control
   - Two-tier: auth middleware + admin email whitelist
   - Dev mode: if ADMIN_EMAILS empty, allows all authenticated users
   - Prod mode: strict email whitelist enforcement

7. Async Operations
   - POINTS_QUEUE: point grants, bonuses
   - ACHIEVEMENTS_QUEUE: achievement unlocks
   - NOTIFICATIONS_QUEUE: user notifications

8. Real-time Connections
   - WebSocket via Durable Objects
   - GameRoom: per-event game state
   - GroupRoom: per-group chat/notifications
